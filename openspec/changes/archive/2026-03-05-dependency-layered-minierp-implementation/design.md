## Context

miniERP 当前处于“设计完整、实现骨架化”阶段：`designs/` 已定义业务流程、页面模板（T1/T2/T3/T4）与证据双层模型，但 `apps/server` 与 `apps/web` 仍以 scaffold 为主。若直接按页面或模块并行开工，会因 shared 契约、数据库模型、租户隔离、库存过账一致性等底层依赖未冻结而频繁返工。

本次设计目标是把实施顺序从“按模块平铺推进”改为“按依赖层级自底向上推进”，并提供可并行且最小阻塞的解锁机制。

约束条件：
- 技术栈固定为 Next.js 15 + React 19、NestJS 11、TypeScript、Bun workspace。
- 业务规则固定：单据号规则、decimal.js 金额语义、状态流转可审计。
- 设计复用固定：T1-T4 模板体系、证据双层模型。
- 多租户与库存一致性必须在底层先落实，不能后补。

## Goals / Non-Goals

**Goals:**
- 建立“依赖层级驱动”的实施架构：基础层 -> 领域后端层 -> 前端装配层。
- 明确可并行工作流与解锁条件，让不同 stream 在不互相阻塞的前提下推进。
- 固化 shared 契约、数据库逻辑模型、状态机与索引策略，作为后续实现唯一基线。
- 将库存过账一致性、幂等、防负库存和证据双层模型作为强约束前置。
- 为后续 tasks 拆解提供可执行的依赖图，而非抽象建议。

**Non-Goals:**
- 本阶段不实现业务代码，不引入新运行时框架。
- 不在本阶段完成具体 UI 细节还原或性能压测实现。
- 不在本阶段做微服务拆分；维持模块化单体策略。
- 不扩展超出当前目标范围的新增业务域（如财务总账深度建模）。

## Decisions

### 决策 1：采用“三层依赖推进模型”替代“按模块平铺推进”

**方案选择**
- 选中：按依赖层级推进
  1) L0 基础层：shared 契约、数据库逻辑模型、多租户与审计基础能力
  2) L1 领域后端层：采购/销售/库存/证据/盘点等领域命令与查询
  3) L2 前端装配层：模板组件、SDK/BFF、页面联调、E2E
- 备选：按业务模块（如 SKU、采购、销售）各自端到端推进

**选择理由**
- 依赖层推进可把全局约束一次性冻结，减少跨模块返工。
- 上层工作以“已冻结契约”为输入，避免字段和状态语义漂移。

---

### 决策 2：L0 必须先冻结 shared 与数据库逻辑模型

**方案选择**
- 选中：L0 先冻结以下内容
  - `DocumentType/DocumentStatus` 与状态机映射
  - API 包络、分页、错误码语义
  - 逻辑实体、关系、索引、复合唯一（tenant 维度）
  - 审计字段与幂等字段规范
- 备选：L1 开发过程中逐步补齐契约与表结构

**选择理由**
- 逐步补齐会导致 L1 与 L2 同步等待，且容易产生不兼容变更。
- 先冻结可让后端与前端并行消费统一契约。

---

### 决策 3：后端采用“命令 REST + 查询 REST/GraphQL”分层策略

**方案选择**
- 选中：命令与查询职责分离
  - 命令接口：创建、确认、过账、作废、冲销、上传会话
  - 查询接口：列表、详情、聚合、流水、统计
- 备选：全部 REST 或全部 GraphQL

**选择理由**
- 与既有架构意图一致，且有利于读写路径独立演进。
- 命令链路可严格约束事务边界与幂等；查询链路可优化聚合能力。

---

### 决策 4：库存一致性作为 L0/L1 间“硬闸门”

**方案选择**
- 选中：库存相关能力在所有出入库业务上线前必须满足：
  - 幂等键机制
  - 事务内原子过账
  - ledger 事实源
  - 防负库存双防线
  - 冲销补偿策略
- 备选：先让业务流程跑通，库存一致性后补

**选择理由**
- 库存污染代价高且难回溯，必须先建正确性底座。

---

### 决策 5：证据系统按“asset/link 双实体 + document/line 双作用域”落地

**方案选择**
- 选中：
  - `evidence_asset` 负责对象元数据与状态
  - `evidence_link` 负责业务绑定（单据级/行级）
- 备选：单表附件模型（仅 document 级）

**选择理由**
- 单表难支持行级追溯和争议定位，与现有设计要求冲突。

---

### 决策 6：前端先模板层后页面层，页面不得绕过模板契约

**方案选择**
- 选中：T1-T4 模板与 Evidence 组件先行，页面作为配置与数据编排装配。
- 备选：逐页独立实现，后续再抽模板。

**选择理由**
- 先模板可显著降低页面差异化和重复开发，确保联调输入统一。

## Risks / Trade-offs

- [L0 设计冻结过慢] → Mitigation：设置“冻结窗口 + 变更评审门禁”，超过窗口仅允许兼容性变更。
- [能力拆分过细导致管理成本上升] → Mitigation：以“可独立验收 + 有清晰输入输出”为拆分标准，避免过度原子化。
- [L1 并行流存在接口耦合冲突] → Mitigation：所有跨流接口先发布契约草案并建立契约测试。
- [L2 前端等待后端接口] → Mitigation：先使用 shared DTO + mock server + SDK 桩实现模板装配，后切真实接口。
- [多租户/RLS 落地影响查询性能] → Mitigation：tenant 前缀索引 + explain 基线 + 热路径专项优化。
- [证据异步处理导致最终一致窗口] → Mitigation：引入证据状态机与 UI 状态提示（处理中/可用/拒绝）。

## Migration Plan

1. **阶段 A：冻结底层依赖（L0）**
   - 输出 shared 契约冻结版与数据库逻辑模型冻结版。
   - 定义租户隔离、审计、幂等、状态机统一规范。
   - 产出解锁条件 A：`L0-READY`。

2. **阶段 B：并行推进后端领域能力（L1）**
   - 按“核心链路流 + 支撑能力流”并行。
   - 核心链路：purchase/inbound/sales/outbound/inventory/stocktake。
   - 支撑能力：evidence/iam/tenant/audit/platform。
   - 产出解锁条件 B：`L1-API-READY`。

3. **阶段 C：前端模板与页面装配（L2）**
   - 先模板组件、SDK/BFF、hooks，再页面路由装配。
   - 先打通 GRN/OUT/Stocktake，再扩展 SKU/PO/SO/设置。
   - 产出解锁条件 C：`L2-E2E-READY`。

4. **阶段 D：联调与收口**
   - 契约回归、状态流转回归、跨租户安全回归。
   - 核心流程 E2E 收敛后进入 apply。

5. **回滚策略**
   - 文档与契约层：版本回退到上一个冻结 tag。
   - 数据层：采用前向兼容迁移，禁止破坏性回滚；通过“停用新路径 + 保留旧路径”回退。
   - API 层：新增字段默认可选，删除字段需经历弃用窗口。

## Open Questions

- `specs` 工件阶段是否按“每个 capability 一个 spec”严格拆成 7 个 spec，还是先按 L0/L1/L2 聚合后再细分？
- tasks 是否采用“依赖图 + owner/stream + DoD”格式（推荐）以便直接并行分工？
- GraphQL 查询能力在本变更中是否先覆盖工作台聚合查询，还是仅定义接口边界到下一变更实现？
- 证据处理链路中的病毒扫描在本次任务范围内定义为必选还是可配置可选？
- 多币种字段本次是否只做 schema 预留，不引入汇率换算规则？
