## 2. designs/ui/minierp_page_spec.md 的错误设计

### 2.1 问题 1："模板" vs "骨架" 的语义混淆

**原文**：
```
## A. 通用页面模板（4 种布局）

### T1) 概览页模板 OverviewLayout
**用途**：只负责"指路"，不负责"干活"。
**页面结构**
* 左侧：主导航（模块列表）
* 顶部：页面标题 + 搜索框（按该模块搜索）+ 主按钮（新建/过账）
* 主区 A：KPI 卡片区（3~4 张）
...
```

**问题**：
1. ❌ 用词是"**模板**"而不是"**骨架约束**"
2. ❌ 详细规定了页面结构（左侧、顶部、主区 A/B、右侧、底部）
3. ❌ 暗示 agent 可以按模板批量生成页面

**导致的错误实现**：
- agent 把 T1/T2/T3/T4 理解成"模板系统"
- 创建 `erp-page-assemblies.tsx` 作为"模板生产工厂"
- 用配置驱动页面生成，而不是逐页实现

**应该怎么写**：
```
## A. 页面骨架约束（4 种 family）

### T1 = Hub / Dashboard family
**约束**：页面骨架包含 KPI 展示 + 导航入口
**不约束**：具体布局、组件选择、交互细节
**必须**：复刻 pencil 设计稿的具体形态
```

### 2.2 问题 2：暗示"批量生成"是可以接受的

**原文**：
```
## B. 页面清单（共 30 页）
...
```

**问题**：
- ❌ 列出了 30 个页面，暗示可以批量生成
- ❌ 没有明确"每个页面必须独立实现"
- ❌ 没有明确"复用只允许在 primitives / shells 层"

**导致的错误实现**：
- agent 创建 `RoutePlaceholderPage` 批量占位
- agent 用 re-export 批量复用 `settings/master-data/page.tsx`
- 页面数量快速增长（4 → 43 → 113），但质量不达标

**应该怎么写**：
```
## B. 页面清单与实现顺序

**禁止批量生成**：每个页面必须独立实现，逐页验证。

**实现顺序**：
1. 先完成 3 个核心页面（/workspace、/mdm/items、/reports）
2. 验证设计一致性 + 数据联调 + 测试通过
3. 再继续下一批

**完成标准**：
- 设计一致性：与 pencil 设计稿一致
- 数据联调：真实调用 BFF，不是 mock
- 测试通过：有对应测试

**禁止**：
- ❌ 批量生成页面
- ❌ re-export 其他页面的 page.tsx
- ❌ 使用 RoutePlaceholderPage
```

## 3. designs/ui/miniERP_design_summary.md 的错误设计

### 3.1 问题 1："模板覆盖率" 误导

**原文**：
```
## 六种布局模板全部覆盖 ✅

| 布局 | 已实现页面 | 核心特征 |
|------|-----------|----------| 
| **T1 概览页** | `/ 工作台首页` · `/skus/overview` ... | ... |
| **T2 列表工作台** | `/skus` · `/settings/master-data` ... | ... |
...
```

**问题**：
1. ❌ 用"✅"标记"已覆盖"，暗示模板比设计稿重要
2. ❌ 没有明确"模板覆盖 ≠ 页面完成"
3. ❌ 暗示 agent 可以"套模板"就完成页面

**导致的错误实现**：
- agent 满足于"模板覆盖"
- 不去逐页复刻设计稿
- 页面"看起来像模板"但不是设计稿的形态

**应该怎么写**：
```
## family 骨架覆盖情况

**注意**：family 只是骨架约束，不是完成标准。

| family | 已有页面 | 设计稿复刻完成 | 数据联调完成 | 测试通过 |
|--------|----------|----------------|--------------|----------|
| T1 | 5 | 1 (20%) | 0 (0%) | 0 (0%) |
| T2 | 10 | 2 (20%) | 1 (10%) | 0 (0%) |
...

**完成标准**：
- ✅ 设计稿复刻
- ✅ 数据联调
- ✅ 测试通过
```

### 3.2 问题 2："换数据+换字段" 的复用策略

**原文**：
```
> [!TIP]
> 6 种布局模板均已有代表性实现。剩余 24 页都是"换数据+换字段"的复用变体，开发时直接套用对应母版组件即可。
```

**问题**：
1. ❌ 暗示"换数据+换字段"就够了
2. ❌ 没有强调"每页必须独立实现"
3. ❌ 没有强调"设计稿复刻"的重要性

**导致的错误实现**：
- agent 真的就"换数据+换字段"
- 不去复刻设计稿的具体形态
- 页面同质化，没有业务特色

**应该怎么写**：
```
> [!WARNING]
> family 只是骨架约束。每个页面必须：
> 1. 复刻 pencil 设计稿的具体形态
> 2. 实现页面级的 view model
> 3. 通过设计一致性验证
> 
> 禁止"换数据+换字段"式的批量生成。
```

## 4. AGENTS.md / CLAW.md 的错误设计（重置前）

### 4.1 问题 1：缺少明确的"完成标准"

**原文（AGENTS.md）**：
```
默认策略：
- 已有设计稿映射的正式页面，优先实现 page-level view。
- 只复用 primitives / shells / 局部业务块，不要新造一次性大模板，也不要复活万能 assembly。
```

**问题**：
1. ❌ 没有定义什么是"实现 page-level view"
2. ❌ 没有明确"完成"的标准
3. ❌ 没有可检查的检查点

**导致的错误实现**：
- agent 创建了 page-level view 文件，但里面是空壳
- agent 认为文件存在就是完成
- 没有验证设计一致性、数据联调、测试

**应该怎么写**：
```
## 完成标准（必须全部满足）

页面完成必须满足以下 5 项：

1. **设计一致性**
   - [ ] 页面结构与 pencil 设计稿一致
   - [ ] 没有设计稿未映射的元素
   - [ ] 通过设计一致性审查

2. **数据联调**
   - [ ] 真实调用 BFF + 后端
   - [ ] 不是静态 mock 数据
   - [ ] 错误处理完整

3. **测试通过**
   - [ ] 单元测试覆盖核心逻辑
   - [ ] 集成测试通过
   - [ ] E2E 测试（如适用）

4. **代码审查**
   - [ ] 通过 PR review
   - [ ] 没有遗留的 TODO/FIXME
   - [ ] 代码符合工程红线

5. **文档同步**
   - [ ] 四文档一致性检查通过
   - [ ] 页面状态台账已更新

**禁止的"假完成"**：
- ❌ 路由可访问但页面是 re-export
- ❌ 页面有壳但数据是静态 mock
- ❌ 使用 RoutePlaceholderPage
- ❌ 复用其他业务路由的 page.tsx
```

### 4.2 问题 2："family 治理" 描述不清晰

**原文（CLAW.md）**：
```
2. **前端采用设计稿驱动 + family 治理**
   - 页面级 view：`apps/web/src/components/views/erp/`
   - family shells：`apps/web/src/components/shells/erp/`
   - primitives：`apps/web/src/components/primitives/erp/`
   - 语义配置与 legacy：`apps/web/src/components/business/`
```

**问题**：
1. ❌ 没有明确 family 的定义
2. ❌ 没有明确"family 只约束骨架"
3. ❌ 没有明确"family 不约束具体 UI"

**导致的错误实现**：
- agent 在 family 之上又创建了 `family-variants`
- family 变成了"模板分类法"
- family 承载了太多业务逻辑

**应该怎么写**：
```
## family 定义（只约束骨架，不约束 UI）

| family | 骨架约束 | 不约束的内容 |
|--------|----------|--------------|
| T1 | 页面包含 KPI + 导航入口 | 具体布局、组件、交互 |
| T2 | 页面包含列表 + 筛选 | 表格列、筛选字段、操作按钮 |
| T3 | 页面包含详情展示 | 信息分区、Tab 内容、操作按钮 |
| T4 | 页面包含步骤流转 | 步骤数量、表单字段、校验规则 |

**关键规则**：
- ✅ family 只约束骨架
- ✅ 具体 UI 必须复刻 pencil 设计稿
- ❌ family 不承载业务逻辑
- ❌ family 不定义具体 UI
```

### 4.3 问题 3：缺少"禁止假完成"的明确约束

**原文**：无

**问题**：
- ❌ 没有明确禁止 re-export
- ❌ 没有明确禁止 RoutePlaceholderPage
- ❌ 没有明确禁止静态 mock 数据

**导致的错误实现**：
- 大量使用 re-export（`export { default } from '@/app/(dashboard)/settings/master-data/page'`）
- 大量使用 RoutePlaceholderPage
- 大量使用静态 mock 数据

**应该怎么写**：
```
## 禁止的"假完成"

以下情况**不算完成**：

- ❌ **re-export**：`export { default } from '@/app/.../page'`
- ❌ **RoutePlaceholderPage**：使用占位组件
- ❌ **静态 mock**：数据是硬编码的静态数据
- ❌ **空壳页面**：只有文件，没有实际实现
- ❌ **模板套用**：只换数据不换设计

**检查方法**：
```bash
# 检查 re-export
grep -r "export.*from.*page" apps/web/src/app

# 检查 RoutePlaceholderPage
grep -r "RoutePlaceholderPage" apps/web/src/app

# 检查静态数据
grep -r "const.*=.*\[" apps/web/src/components/views
```
```

## 5. docs/plans/2026-03-07-erp-page-reconstruction-design.md 的问题

### 5.1 问题 1：为什么在 03-07 才提出"停止万能装配器"？

**原文**：
```
当前运行时大量 dashboard 页面并不是按 pencil 设计稿逐页实现，而是：

- route 直接转发到 `settings/master-data` 或 placeholder
- 或通过旧 assembly / template config 生成"像模板"的页面
```

**问题**：
1. ❌ 说明之前的文档本身就允许万能装配器
2. ❌ 说明之前的文档没有明确"禁止万能装配器"
3. ❌ 说明之前的文档没有可执行的约束

**导致的错误实现**：
- 在 03-07 之前，agent 大量使用万能装配器
- 在 03-07 之后，agent 仍然继续使用（因为文档是"后加的"）

**应该怎么写**：
从项目初始化（03-01）开始，就应该明确：

```
## 工程红线（从第一天开始）

13. **禁止新的万能页面装配器**
    - ❌ 不允许创建类似 `erp-page-assemblies.tsx` 的文件
    - ❌ 不允许用配置驱动页面生成
    - ✅ 每个页面必须独立实现
    - ✅ 复用只允许在 primitives / shells 层
```

### 5.2 问题 2："视觉优先"策略被误解

**原文**：
```
用户已明确确认本轮采用 **视觉优先** 的推进方式：

- 先把页面按 `images/source/miniERP-pencil-opus4.6.pen` 复刻正确
- 先完成 page-level view 的正式页面形态
- 后续再补充逻辑、BFF 收敛、测试链路
```

**问题**：
1. ❌ "视觉优先"被误解为"只要视觉"
2. ❌ "后续再补充"被无限推迟
3. ❌ 没有明确"视觉完成后多久必须补充逻辑"

**导致的错误实现**：
- agent 完成视觉后，就认为页面完成
- 逻辑、BFF、测试一直没有补充
- 最终导致"视觉完成"但"业务不闭环"

**应该怎么写**：
```
## 视觉优先策略（明确时间窗口）

**视觉优先 ≠ 只要视觉**

**正确理解**：
1. 视觉是第一步，不是最后一步
2. 视觉完成后 **必须在 3 天内** 补充逻辑
3. 视觉 + 逻辑 + 测试才是完成

**时间窗口**：
- Day 1-2：视觉复刻
- Day 3：数据联调
- Day 4：测试补齐
- Day 5：代码审查 + 文档同步

**禁止**：
- ❌ 视觉完成后就认为页面完成
- ❌ "后续再补充"没有时间限制
- ❌ 只做视觉不做逻辑
```

## 7. 总结：文档本身的设计错误

### 7.1 根本问题

**文档本身就是按照"模板系统"的思想设计的**，而不是按照"骨架约束 + 设计稿复刻"的思想。

### 7.2 具体错误

1. **用词错误**："模板" vs "骨架"
2. **重点错误**：强调"模板覆盖"而非"设计稿复刻"
3. **约束缺失**：缺少"完成标准"、"禁止假完成"
4. **时间模糊**："后续再补充"没有时间窗口
5. **不可检查**：约束无法通过脚本检查

### 7.3 教训

1. **文档的第一性原理**：文档的目的是指导开发，防止偏离
2. **约束必须可检查**：所有约束必须可以通过 CI/脚本检查
3. **完成必须明确定义**：不能靠"感觉"，必须有检查清单
4. **文档必须先于代码**：从第一天开始就要有正确的约束

---

## 8. 建议：文档应该怎么写

### 8.1 结构

1. **第一性原理**：明确文档的目的
2. **约束**：明确允许什么、禁止什么
3. **完成标准**：明确的检查清单
4. **时间窗口**：明确的时间限制
5. **可检查**：提供检查脚本

### 8.2 用词

- ❌ "模板" → ✅ "骨架约束"
- ❌ "覆盖" → ✅ "复刻"
- ❌ "优先" → ✅ "第一步（3 天内完成第二步）"
- ❌ "复用" → ✅ "只允许复用 primitives / shells"

### 8.3 约束

```
## 约束（必须遵守，违反 PR 拒绝）

### 允许
- ✅ 复刻 pencil 设计稿
- ✅ 复用 primitives / shells
- ✅ 独立实现每个页面

### 禁止
- ❌ 批量生成页面
- ❌ re-export 其他页面
- ❌ 使用 RoutePlaceholderPage
- ❌ 静态 mock 数据
- ❌ 万能装配器

### 检查
- CI 门禁：grep 检查禁止项
- 人工审查：设计一致性
- 自动化：页面状态台账
```
