## 3. 根因分析

### 3.1 根本原因：生成式开发的陷阱

**核心问题**：从设计图直接生成代码，速度快但缺乏业务深度。

证据：
```
bcde51d: feat(web): 从设计图生成 miniERP 前端页面代码 (Nordic Brutalist Light)
cd25993: feat(web): 复刻 .pen 设计稿全部26个页面
```

**后果**：
- 生成的页面是"模板壳 + 种子数据"
- `feature-audit-2026-03-05.md` 明确指出："页面覆盖率高，但业务完成度偏低"
- 大量页面"存在但未联调"

### 3.2 次要原因

#### 3.2.1 万能装配器依赖

工程红线明确禁止：
> `WorkbenchAssembly` / `OverviewAssembly` 仅允许用于 legacy fallback、临时页、未重构页。

但实际使用广泛，导致：
- 页面结构同质化
- 无法针对具体业务定制
- 最终需要全部重写

#### 3.2.2 多 Agent 并行开发规则漂移

两个主要贡献者同时开发：
- `xuxunjian`：大量 feat(web) 提交
- `Jcateye`：PR 合并和 review

问题：
- 03-05 一天内有大量提交
- 规则文档（CLAW.md）在 03-06 才补齐"工程红线"
- 四文档同步可能不及时

#### 3.2.3 视觉优先策略的执行偏差

`2026-03-07-erp-page-reconstruction-design.md` 明确：
> 先视觉复刻，再补逻辑与测试

但实际：
- 视觉复刻完成（37 页）
- 逻辑一直没补上
- 最终选择重置而非继续修补

## 5. 教训总结

### 5.1 生成式开发的风险

**问题**：AI 生成代码速度快，但：
- 缺乏业务上下文
- 生成的是"壳"而非"肉"
- 需要大量后续补全

**教训**：
- 生成后立即验证业务闭环
- 不要批量生成后批量修补
- 每个页面生成后立即联调

### 5.2 规则先行的必要性

**问题**：工程红线在 03-06 才写入 CLAW.md，而开发从 03-01 开始。

**教训**：
- 规则必须在开发前建立
- 多 agent 协作必须先冻结规则
- 四文档同步必须作为 PR 门禁

### 5.3 视觉优先的正确姿势

**问题**："视觉优先"被误解为"只要视觉"。

**正确理解**：
- 视觉是第一步，不是最后一步
- 视觉完成后立即补逻辑
- 每个 page-level view 必须有对应的 view-model

### 5.4 早期止损的重要性

**问题**：03-05 的 feature-audit 已经指出问题，但重置在 03-11。

**教训**：
- 定期（每日）检查"业务闭环率"
- 当闭环率 < 50% 时停止新页面开发
- 优先补齐已有页面的逻辑

## 6. 预防措施

### 6.1 开发流程改进

1. **增量生成，即时验证**
   - 每次生成 1-3 个页面
   - 立即联调 BFF + 后端
   - 通过后才继续下一批

2. **规则门禁**
   - PR 必须检查四文档一致性
   - 新页面禁止使用万能装配器
   - BFF 调用必须真实（禁止静态数据）

3. **业务闭环率监控**
   - 每日统计：已联调页面 / 总页面
   - 阈值：< 70% 时禁止新增页面

### 6.2 Zoe 工作流改进

1. **任务派发时明确约束**
   - 每个任务必须包含"首个可交付件"
   - 必须明确"作用域"和"checkpoint"
   - Codex 适合执行，Gemini 适合审计

2. **定期健康检查**
   - 每 10 分钟检查 agent 状态
   - 每 4 小时检查业务闭环率
   - 发现 stalled agent 立即干预

3. **文档先行**
   - 规则变更必须先更新 CLAW.md
   - 四文档同步作为 PR 必要条件
   - 设计文档与代码同步更新
